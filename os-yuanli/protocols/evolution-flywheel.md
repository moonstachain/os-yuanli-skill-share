# Evolution Flywheel

每个任务闭环 = 一次进化机会。

这个协议规定一次进化如何被识别、记录、沉淀，以及**什么不应该被沉淀**。

## 核心原则

> 每做一次任务，系统都应该比上一次略强。

进化不是任务的**附赠**，而是任务的**产物之一**。

## Evolution Note 契约

每个任务在 `closed` 前必须填写 `## Evolution` 段落，至少包含：

- `what was gained`
  这一轮学到或验证了什么？
- `what was wasted`
  浪费在哪里？返工、误入歧途、算力空转、人工白干？
- `what should improve next`
  下一轮应该怎么优化？具体到**操作**，而不是感叹。

**空 Evolution 不允许**——哪怕写一句"本轮无新增进化"，也是合法状态，不允许跳过。

## 升级三通道

Evolution 观察到的改进点可以走三条升级通道之一：

### 1. 沉淀为新 Skill

**触发条件**：

- 相同模式在 ≥ 3 个任务里成功复用
- 当前能力缺失，手动重复代价明显

**不触发**：

- 只跑过 1 次的"偶然成功"
- 业务变更概率高的临时方案

### 2. 沉淀为新 Guide

**触发条件**：

- 观察到某类失败会反复发生
- 当前规则无法阻止该失败

**不触发**：

- 单次偶发错误（修了就行）
- 可以通过 sensor 事后检测的问题（不要在前馈里加复杂度）

### 3. 沉淀为新 Template / 认知条目

**触发条件**：

- 模式未成熟到封装 skill，但已经有复用价值
- 发现了新的失真点、认知盲区

## 不要升级的情况

**常见陷阱**——这些都不该被"进化":

| 现象 | 为什么不要进化 |
|---|---|
| 临时 annoyance | 抱怨 ≠ 结构性问题 |
| 单次巧合成功 | 可能只是运气好 |
| 用户一次性偏好 | 偏好会变，不要写进规则 |
| 过度通用化的感悟 | 没有具体操作的"洞察"是负债 |

进化的硬门槛是**能不能立刻被下次任务消费**。

## 合流与去重

进化产物会快速膨胀。必须定期合流：

- 两条 guide 讲同一件事 → 合一条
- 两个 skill 90% 重叠 → 合并或上下位化
- 失去语境的 note → 归档或删除

合流本身是一个任务，走完整 OS 协议（不是"整理一下"这么随便）。

## 与 Guides / Sensors 的关系

```
    Sensor 发现偏差
          │
          ▼
    Evolution Note
          │
    ┌─────┼─────┐
    ▼     ▼     ▼
  新Skill 新Guide 新Template
          │
          ▼
    下轮任务消费它们
          │
          ▼
    Sensor 再次检测
    → 闭环
```

Evolution 是 Sensors 与 Guides 之间的**中间件**。

## Inclusion

- Evolution Note 最小字段
- 三条升级通道的触发与不触发条件
- 不该进化的反模式清单
- 与 Guides / Sensors 的位置关系

## Exclusion

本协议**不规定**：

- Evolution Note 存哪里（任务正文内 / 单独文件 / 数据库）
- 升级审批流程（单人 setup 可能无审批，团队 setup 可能要走 PR）
- 合流频率
- 是否有自动化工具做去重

这些都是使用者自己的工程选择。
