# Concepts · 为什么是这套结构

面向想理解"**为什么**要这样设计"的读者。

## 一句话总结

> `os-yuanli` = `治理OS × 工作OS` = 在**任何**复杂任务上，先问"值不值得进入、怎么赢、怎么闭环"，再谈执行。

## 四层架构为什么是这四层

```
Tier 0 · Kernel       方法内核（六判断 / 3×3 / 三层 gate）
Tier 1 · Capability   协议包（任务 / 路由 / brain / 回流 / 进化 …）
Tier 2 · Adapter      使用者自填（themes / runtime / bridges）
Tier 3 · Reference    实例化样板（examples/）
```

### 为什么要分 Kernel 与 Capability

**Kernel 是"该做什么判断"**：
六判断、3×3 治理线、主题/策略/执行三层 gate。
这些是**普适的**——任何一个多阶段任务都适用。

**Capability 是"判断背后的协议细节"**：
任务 `.md` 长什么样、状态怎么流、brain 分几 tier……
这些是**操作级**的——不同团队可能有不同实现。

把二者分开，是为了让 **kernel 永远独立可用**：你甚至可以不加载 protocols，只用 kernel 接管一个任务。

### 为什么要有 Adapter

**Capability 只定义规则，不定义内容**。

比如 `task-protocol.md` 规定"任务必须有 theme 字段"——但**具体主题叫什么**，完全是使用者的自由度。

如果把具体主题写进 Capability，就等于把作者的战场塞进协议里，新用户被迫继承，产品立刻变重。

所以 Adapter 做一件事：**填补 Capability 留下的 Exclusion 段**。

### 为什么要有 Reference

有了 Kernel + Capability + Adapter，理论上任何人都能配出自己的 OS。

但**空模板让人没安全感**。Reference 给 3 个参考实例——minimal（最小）、team-engineer（常见工程团队）、hay-setup（作者丰富实例）——让新用户能找到最接近自己处境的那个抄一个初稿。

## 六判断为什么是这六个

六判断不是"做什么"的清单，是"**在开始做之前必须想清楚的六个问题**"：

| 判断 | 防的是什么 |
|---|---|
| 自治判断 | 越权 / 浪费不该由你做的事 |
| 全局最优判断 | 局部勤奋掩盖整体偏航 |
| 能力复用判断 | 重复造轮子 |
| 验真判断 | 假完成（自己说做完了就算完了）|
| 进化判断 | 用完就扔，系统不变强 |
| 当前最大失真 | 在错误假设上推进 |

每一条都是一种常见的**失败模式**的预防剂。

## 三层 gate 为什么是这三层

```
主题层 —— 值不值得进入这场仗？
  ↓
策略层 —— 怎么赢这场仗？
  ↓
执行层 —— 跑一条可验真的链路
```

为什么**不能跳**：

- 跳过主题 gate → 做了一件不该做的事，时间回不来
- 跳过策略 gate → 执行起来很忙，但方向错了
- 不设执行 gate（不验真）→ 自己骗自己"做完了"

这三层对应经典的决策论三问：选题 / 方法 / 落地。

## 3 × 3 治理矩阵的用法

|  | 递归进化 | 全局最优 | 人类友好 |
|---|---|---|---|
| **主题层** | 会不会沉淀长期能力？ | 是不是当前最大战场？ | 让人更聚焦还是更散？ |
| **策略层** | 能不能变成复用框架？ | 解核心矛盾还是修局部？ | 减少返工还是增加审校？ |
| **执行层** | 这条链路越跑越稳吗？ | 推动结果还是制造进度幻觉？ | 真实减轻体力 / 算力吗？ |

在每一层都过这三条线，绝大部分"看起来很忙但没产出"的任务会在某个格子卡住。

这不是一次性 checklist，是每次任务的 live filter。

## 策略层次阶梯

不是所有"策略"都配得上叫策略：

```
数据 → 特征 → 观点 → 洞察 → 全局最优
```

- **数据**：原始事实
- **特征**：抽出了可命名的模式
- **观点**：有倾向，但还没解释为什么
- **洞察**：揭示了别人没看见的结构性机制
- **全局最优**：在系统层面给出了取舍与方向

策略 gate 的关键问题："你现在在哪一层？该不该跃迁到上一层？"

大多数被拒绝的策略其实停在了**观点**——能说一段，但不足以赢。

## 验真（evidence）为什么独立成段

因为人（和 AI）都有"差不多就行了"的天性。

一旦写在任务模板里成为硬字段，就成了**协议义务**——不填它，状态不能 `closed`。

这是把 AI 对齐问题变成文件系统问题——任务 `.md` 少一段，审计就拦下来。比靠"自律"可靠得多。

## 进化（evolution）为什么也是硬字段

同样原因：**闭环必须留下可消费物**。

如果一个任务做完了什么都没沉淀，系统就永远不会变强——第 N+1 次做同类任务，你还是从零开始。

Evolution 段位不能空——这一条硬约束让 OS 真的会进化。
