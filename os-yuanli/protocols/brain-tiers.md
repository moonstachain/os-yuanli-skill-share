# Brain Tiers

本协议规定如何把任务分派给不同 tier 的模型，以及成本约束规则。

## 三 tier 结构

### A · Strategy Brain · 策略大脑

**用途**：

- 方法论设计
- 复杂问题拆解
- 非结构化判断 / 创造性发散
- 主题 gate 与策略 gate 的判断

**代表模型**：Claude Opus、GPT 顶级推理模型、专用 reasoning model。

**进入条件**：Sonnet 级别无法替代的**判断**或**创造**任务。

### B · Execute Brain · 执行大脑

**用途**：

- 结构化执行（写代码、写文案、跑流程）
- 数据处理与格式转换
- API 编排与自动化
- Skill 开发与测试

**代表模型**：Claude Sonnet、Codex、同级执行模型。

**进入条件**：任务拆解已经完成，路径清晰，需要高质量结构化输出。

默认主力——除非触发升级/降级条件。

### C · Knowledge Brain · 知识大脑

**用途**：

- 已有知识库检索与整理
- 文件分类 / 去重 / 归档
- 健康巡检 / lint
- 批量的机械转换

**代表模型**：Claude Haiku、轻量模型、规则脚本、传统 NLP 工具。

**进入条件**：任务本质是"查 / 分 / 整"，而非"判 / 创 / 生"。

## 成本约束原则

按优先级从高到低：

1. **能用确定性脚本解决的，不引入任何 LLM**
   任何需要被严格复现的 step（格式转换、校验、lint），应该优先脚本化。
2. **能用 C tier 解决的，不上 B tier**
3. **能用 B tier 解决的，不上 A tier**
4. **能用 A tier 一次决策、B tier 批量执行 的，不要全程 A tier**

## 组合模式（Compose）

现实任务大多需要多个 tier 协作：

### 策略 → 执行（串行）

A 产出决策 / 大纲 / 规范，B 按规范批量产出。
示例：A 设计一份 PRD 的章节结构 → B 填每一章的内容。

### 执行 ← 知识（并行）

B 在执行中按需调用 C 做检索，不让 B 自己生成事实。
示例：B 写文章时，C 提供事实引用。

### 闭环仲裁（策略兜底）

B 执行遇到超出预期的结构性问题 → `escalated` 回 A。
A 不做底层执行，但做路线仲裁。

## 升级与降级

### 升级触发条件（C→B 或 B→A）

- 出现结构性歧义，B 连续两次输出低质量
- 涉及高风险决策（钱、信任、发布）
- 需要跨多个知识域整合判断

### 降级触发条件（A→B 或 B→C）

- 任务被拆解后发现实际是"执行"或"查询"
- 反复在同一 tier 重跑浪费成本
- 路径已经模板化（跑过 3 次以上）

## Inclusion

- 三 tier 的定义与分工
- 成本约束四条规则
- 升级 / 降级触发条件

## Exclusion

本协议**不规定**：

- 具体用哪一家的模型（可以是 Claude / OpenAI / Gemini / 本地模型的任意组合）
- 每个 tier 用哪个具体模型（使用者可以在 `runtime.yaml` 里声明）
- 是否允许混用多家厂商
- 如何核算成本（使用者自己的账单系统）

只约束"用什么级别的脑子"，不约束"用谁家的脑子"。
