# Router · Five Axes

当一个任务进入执行层，必须选择"用什么"来执行（skill / tool / MCP / runtime）。

本协议定义选择时的 5 个排序轴。

## 为什么需要显式的五维

没有显式排序规则时，路由会漂向：

- 偏好工具（最熟的那个）
- 最近用过的工具（最近路径偏置）
- 最"聪明"的模型（成本失控）

这些都不是**任务层面的最优**。

## The Five Axes（按优先级从高到低）

### 1. 任务适配度（Task Fit）

这个能力是不是**真的为这类任务设计**？

- 强：能力的声明描述直接匹配任务族
- 中：能力覆盖任务，但需要少量包装
- 弱：能力"能做"但不是为它设计的

**弱适配 + 高成本 的组合是最常见的路由错误**。

### 2. 验真能力（Verifiability）

执行完能否产出**可验真证据**？

- 强：有确定性输出（文件 / diff / 测试结果）
- 中：有可结构化读取的日志
- 弱：只能凭"看起来完成了"判断

没有验真能力的路径，不得承担 `evidence-required` 任务。

### 3. 成本（Cost）

包括：

- 算力成本（模型 tier）
- 时间成本（同步 vs 异步、重试概率）
- 认知成本（人工介入工作量）

默认规则：

> 能用 Haiku 解决的不用 Sonnet；能用 Sonnet 解决的不用 Opus；能用脚本的不用 Agent。

### 4. 能力复用（Reuse Premium）

如果这个路径跑通一次，能否沉淀出可复用的 skill / template / guide？

- 高复用 = 本次成本摊薄到未来 N 次
- 一次性 annoyance = 不值得引入新工具，即使它"看起来更高效"

### 5. 附加复杂度（Added Complexity）

本次执行会给系统留下多少长期复杂度？

- 新依赖 / 新凭证 / 新 bridge / 新守护进程
- 新配置文件 / 新环境变量
- 新的"只有我懂"的隐性约定

低复杂度优先。

## 裁决顺序

当几个候选路径评分接近时，用这个裁决顺序：

```
任务适配度  >  验真能力  >  成本  >  能力复用  >  附加复杂度
```

前两维是**硬门槛**——适配度极差或不可验真，直接淘汰，不进入后续比较。

## Minimum Router Output

执行层必须显式输出以下三件事（见 `output-contract.md`）：

- 选了哪条路径
- 为什么这条路径赢
- 验真目标是什么

不能只说"用 X 工具"，必须对应到本协议的轴做解释。

## Inclusion

- 五维的定义与优先级
- 硬门槛（前两维）
- 最小 router 输出

## Exclusion

本协议**不规定**：

- 具体有哪些 skill / tool / MCP 可选
- 如何对每个轴打分（使用者可自定义评分表）
- 是否使用专门的 router skill（`skill-router` / 类似概念）
- 多路径并行 vs 串行的策略

这些属于使用者的执行层实现细节。
