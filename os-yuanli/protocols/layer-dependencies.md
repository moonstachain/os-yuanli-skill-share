# Layer Dependencies

本协议定义 OS 的分层结构，以及层间依赖规则。

## 五层结构

```
L4 · Scenario Deliverables  场景交付物
      行业 / 客户场景下的成品
           ↑
L3 · Composable Business Flows  可组合业务流
      多 skill 编排的业务流程
           ↑
L2 · Atomic Capabilities  原子能力
      单一 skill / tool / MCP，做一件具体的事
           ↑
L1 · Strategy Methodology  策略方法论
      如何思考、如何判断、如何组织
           ↑
L0 · Evolution Engine  递归进化引擎
      OS 的内核：六判断、3×3、三层 gate、回流协议
```

L0 是发动机，L1–L4 是变速箱与车轮。

## 各层职责

### L0 · Evolution Engine

- 负责"该不该做 / 该怎么做 / 做完了没"
- 不做具体领域工作
- 本 skill 的 kernel（`SKILL.md` + `references/`）就在这一层

### L1 · Strategy Methodology

- 策略框架（例如品类定位、竞争分析、用户研究的方法论）
- 是 L0 的"特定领域实例"
- 不具体执行任务，只给判断框架

### L2 · Atomic Capabilities

- 具体的 skill / tool：写小红书、做 PPT、跑回测、生成图片
- 可独立被调用
- 内部不再 orchestrate 其他 L2 能力（否则就是 L3）

### L3 · Composable Business Flows

- 编排多个 L2 能力的业务流
- 例如"内容生产流水线"、"DTC 产品发射"、"月度客户复盘"
- 是 L2 的组合，不是新能力

### L4 · Scenario Deliverables

- 面向特定场景 / 客户 / 行业的成品
- 是 L3 的**具体应用实例**
- 通常绑定具体的客户上下文或品牌

## 依赖规则（硬约束）

### 规则 1：只能**向下**依赖

- L4 可以调 L3、L2、L1、L0
- L3 可以调 L2、L1、L0
- 依此类推

**不允许反向依赖**（L1 不能依赖某个具体的 L2 skill 存在）。

### 规则 2：同层依赖需谨慎

- 同层互相调用容易形成环
- 如果 A 和 B 同层且 A 依赖 B，通常意味着 B 应该**上移半层**

### 规则 3：跨多层依赖要记账

- L4 直接调 L0 是允许的，但必须在文档里说明跳过了哪些中间层以及为什么
- 避免"隐式短路"

### 规则 4：每层独立可替换

- 替换某一层不应该让其他层集体崩溃
- 例如：把所有 L2 能力换成另一套工具链，L0/L1 应该仍然成立

这条规则是"通用性"的核心保障。

## 反模式

### 反模式 1：L0 污染

在 kernel（L0）里硬编码某个具体 skill（L2）的名字。

**后果**：kernel 不再通用，只能在有那个 skill 的环境里跑。

**纠正**：把具体名字放到 adapter / bridges 里，L0 只引用**角色**（例如 "execute brain"）。

### 反模式 2：L3 膨胀成 kernel

业务流写着写着开始定义自己的判断框架 / gate / sensor。

**后果**：每条业务流都像一个新的 OS，维护成本爆炸。

**纠正**：通用框架上收到 L0 / L1，业务流只做编排。

### 反模式 3：L4 硬编码

交付物里的文案、客户名、业务数据硬写在 L3 业务流里。

**后果**：换个客户就要改 L3。

**纠正**：L3 只处理结构，L4 负责注入具体内容。

## 与其他协议的关系

- `task-protocol.md` 属于 L0 协议层
- `brain-tiers.md` 也是 L0——定义"角色"，不绑定具体模型
- 具体的 skill 文件（如小红书生成器）属于 L2
- 业务流配置（如内容生产流水线）属于 L3
- 某客户的交付包（如"给 X 品牌做月度内容"）属于 L4

## Inclusion

- 五层定义与职责
- 4 条依赖硬规则
- 3 种反模式与纠正方式

## Exclusion

本协议**不规定**：

- 每一层具体有哪些实现（完全由使用者决定）
- 同一个功能应该放 L2 还是 L3（凭判断，参考规则 2）
- 层与层之间的通信协议（可以是函数调用、消息队列、文件、API）
- 每层的版本管理策略
