# Truth Hierarchy

当多个信息源冲突时（LLM 记忆 vs 用户当场指令 vs 配置文件 vs 本地数据），必须有明确的**真源优先级**。

本协议规定五层真源，以及处理冲突的规则。

## 五层真源（从高到低）

### 1. 本地 artifacts / 当前仓库文件

**这是最高真源**。

- 任何与代码、数据、配置有关的判断，必须以**当前状态**为准
- 不依赖记忆、不依赖之前的对话、不依赖 LLM 内部权重

当 memory 里说 "文件 X 存在"，而实际 repo 里 X 已被删除——以**当前状态**为准，memory 需要更新。

### 2. 用户当场指令

本次对话里的**明示指令**。

- 覆盖任何历史偏好、默认规则、配置文件
- 但不能覆盖层 1 的客观事实（用户说"文件 X 存在"不等于文件 X 真的存在）

### 3. 项目级配置

`CLAUDE.md` / `AGENTS.md` / 项目 README / 项目级 memory 等。

- 适用于**本项目**的默认偏好
- 被层 2 的当场指令覆盖

### 4. Skill / 工具级内置知识

本 skill 和它引用的 references。

- 适用于被本 skill 接管的所有任务
- 被层 3 覆盖（项目可以关闭某个 skill 的默认行为）

### 5. LLM 内部权重

模型在训练中学到的一切。

- 是最后的 fallback
- 任何可以在上层真源中验证的事实，都不允许只凭 LLM 权重给出

## 冲突处理规则

### 层间冲突

**以高层为准**，但必须**显式说明**被覆盖的内容：

> 项目 CLAUDE.md 要求 X，但用户当场说改成 Y。按用户当场指令执行 Y。

不允许"悄悄覆盖"。

### 同层冲突

**向上一层请示**——例如两份项目配置冲突，问用户。

不允许自己选一个。

## 时间语义：什么算"当前"

层 1 与层 3 都可能**随时间漂移**。定义如下：

- 层 1（本地文件）：每次被访问时现读
- 层 3（项目配置）：session 开始时读入，session 内生效
- 层 4（skill）：安装时版本锁定，升级前不变

这意味着：**跨多轮对话做判断时，层 1 必须重新访问，不能凭记忆。**

## 记忆（memory）的定位

memory 系统属于**层 3 和层 4 之间**的长期偏好缓存。

但 memory 永远受制于层 1 的客观事实：

- memory 里的文件路径 → 用前必须确认它还存在
- memory 里的决策原因 → 随时间可能过期
- memory 里的活动总结 → 冻结在写入时刻，不代表"当前"

规则：**memory 可以用来建立假设，不能用来替代事实确认**。

## 对 AI 的硬约束

基于本协议，AI 必须：

1. 对**事实类问题**（文件是否存在、命令输出、测试结果、依赖版本）——现场查，不凭记忆
2. 对**偏好类问题**（喜欢什么风格、默认用哪个工具）——按层 2 > 3 > 4 排序
3. 对**冲突的处理**——显式声明，不隐式选择
4. 对**LLM 权重答案**——只在没有上层真源时使用，并标注不确定性

## Inclusion

- 五层真源定义
- 层间与同层冲突处理规则
- memory 的定位与使用边界
- AI 必须遵守的四条硬约束

## Exclusion

本协议**不规定**：

- memory 的具体实现（文件 / 数据库 / 向量检索）
- 如何做本地文件访问（视使用的 runtime 而定）
- 具体项目配置文件的名字
- 是否允许 RAG 作为层 5 的补强
